🧭 建置方法論🧭 Build Methodology

1我怎麼決定「什麼任務值得做成 Agent」How I decide which tasks become Agents

  • 高頻 × 高耗時且規則可描述High-frequency, time-consuming, with rules you can describe
    ex：每次提案都需查詢多張數據表並總結為洞察e.g. every proposal means querying several data tables and distilling them into insight
  • AI 產出需被驗證AI output that needs checking
    ex：團隊高頻用於提案數據鋪陳，但資料敏感度因人而異，可能給出錯誤資訊e.g. heavily used for proposal data, but sensitivity varies by person and wrong answers can slip through
  • 個人經驗難以與團隊共享Expertise that's hard to share
    ex：資深企劃的提案經驗難高效複用全團隊；跨部門資訊落差需工具快速拉齊共識e.g. senior planners' instincts don't scale across the team, and cross-team gaps need a tool to align fast

2我怎麼規劃「嚴謹的 Agent 共通原則」How I set rigorous, shared Agent principles

  • 數據可溯源、反幻覺Traceable data, no hallucination
    每個數字與聲明都須對應到某次工具實際回傳，否則標「數據缺失」，嚴禁 LLM 自行推算every number and claim must map to a real tool return — otherwise it's flagged "data missing," and the LLM never guesses
  • 確定性運算優先於 LLMDeterministic compute over the LLM
    分類、過濾、日期計算改用 Python／regex（可重現），降低隨機性classification, filtering, and date math run in Python/regex (reproducible), cutting randomness
  • 多源工具編排Multi-source orchestration
    串接 Redash、Superset BigQuery、Dcard API、Google Drive、Slack，並以平行查詢縮短等待wires up Redash, Superset BigQuery, Dcard API, Google Drive, and Slack, with parallel queries to cut the wait
  • 品質與安全護欄Quality & safety guardrails
    業配/成人內容程式化過濾、對外數據模糊化、強制套用品牌視覺，確保產出可直接對外programmatic filtering of sponsored/adult content, blurred external data, enforced brand visuals — so output is client-ready
