🧭 建置方法論🧭 Build Methodology

1我怎麼決定「什麼任務值得做成 Agent」How I decide which tasks become Agents

  • 高頻 × 高耗時且規則可描述High-frequency × time-consuming, with describable rules
    ex：每次提案都需查詢多張數據表並總結為洞察e.g. every proposal requires querying many data tables and summarizing them into insight
  • AI 產出需被驗證AI output must be verifiable
    ex：團隊高頻用於提案數據鋪陳，但資料敏感度因人而異，可能給出錯誤資訊e.g. heavily used for proposal data, but data sensitivity varies by person and errors are possible
  • 個人經驗難以與團隊共享Individual experience is hard to share
    ex：資深企劃的提案經驗難高效覆用全團隊；跨部門資訊落差需工具快速拉齊共識e.g. senior planners' experience is hard to reuse team-wide; cross-team info gaps need a tool to quickly align

2我怎麼規劃「嚴謹的 Agent 共通原則」How I design rigorous common Agent principles

  • 數據可溯源、反幻覺Traceable data, anti-hallucination
    每個數字與聲明都須對應到某次工具實際回傳，否則標「數據缺失」，嚴禁 LLM 自行推算every number and claim must map to an actual tool return; otherwise flagged "data missing" — the LLM may never estimate
  • 確定性運算優先於 LLMDeterministic compute over LLM
    分類、過濾、日期計算改用 Python／regex（可重現），降低隨機性classification, filtering, date math handled in Python/regex (reproducible), reducing randomness
  • 多源工具編排Multi-source orchestration
    串接 Redash、Superset BigQuery、Dcard API、Google Drive、Slack，並以平行查詢縮短等待integrates Redash, Superset BigQuery, Dcard API, Google Drive, Slack — with parallel queries to cut wait time
  • 品質與安全護欄Quality & safety guardrails
    業配/成人內容程式化過濾、對外數據模糊化、強制套用品牌視覺，確保產出可直接對外programmatic filtering of sponsored/adult content, output data obfuscation, enforced brand visuals — output is client-ready
